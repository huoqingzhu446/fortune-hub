## 1. 项目目标

`Fortune Hub` 定位为一个面向微信小程序的多功能内容与测评产品，围绕以下核心能力展开：

- 运势内容：星座、八字、幸运签、幸运指数、幸运物推荐
- 测评能力：性格测评、情绪自检
- 用户成长：历史记录、分享海报、VIP 会员、广告解锁
- 后台运营：内容配置、题库管理、报告模板、广告与会员配置

当前仓库是一个多端工程骨架，已经具备：

- 小程序端基础工程
- 管理端基础工程
- API 服务基础工程
- MySQL / Redis / Docker / Nginx 基础接入

但仍未完成完整业务实现。按当前代码状态看，项目已经从“首页演示骨架”推进到了“首版业务闭环”，小程序端的星座、八字、性格测评、情绪自检都已有第一版页面与接口，但距离可上线产品仍有明显差距。

## 3. 技术架构

### 3.1 目标架构

```text
微信小程序（客户端）
├─ 页面层
├─ 逻辑交互层
└─ 数据缓存层

云函数 / 后端接口服务
├─ 业务接口服务
├─ 核心算法服务
└─ 内容管理服务

数据持久层
├─ 数据库
├─ 文件存储
└─ 配置中心
```

### 3.2 当前仓库实际架构

当前代码仓库采用 `pnpm workspace` 多应用结构：

- `apps/mobile`
  小程序端，基于 `uni-app + Vue 3 + Vite + TypeScript + Pinia`
- `apps/admin`
  管理端，基于 `Vue 3 + Vite + Element Plus`
- `services/api`
  API 服务，基于 `NestJS + TypeORM + MySQL + Redis`
- `deploy/nginx`
  Nginx HTTPS 代理配置

当前并未拆分独立“算法服务”或“配置中心服务”，而是以单个 API 服务承载基础接口能力。

## 4. 数据模型设计

### 4.1 目标数据表

```text
user 用户表
├─ openid
├─ 生日 / 性别
├─ 星座 / 五行
└─ VIP 状态

config 配置表
├─ 星座运势
├─ 八字文案
├─ 测评题库
└─ 幸运物库

order 订单表
├─ 订单号
├─ 用户ID
├─ 支付状态
└─ 开通时长

record 记录表
├─ 用户ID
├─ 测评类型
└─ 结果数据

share 海报表
├─ 分享类型
├─ 生成内容
└─ 生成时间

ad 广告配置表
├─ 激励视频开关
└─ 广告位配置
```

### 4.2 当前实际数据层

当前代码中已经不止一个占位实体，实际接入了以下数据表：

- `assessment_sessions`
  早期联调用的示例表，仍然存在，但已不是主业务表
- `users`
  已用于保存登录用户、生日、性别、星座、简易八字摘要、五行、VIP 状态等
- `records`
  已用于保存性格测评、情绪自检、八字解读等结果历史
- `fortune_contents`
  已用于承载幸运签等内容占位与后续内容扩展

当前结论：

- `users`、`records`、`fortune_contents` 已具备首版业务落库能力
- `config`、`order`、`share`、`ad` 等目标表仍未落地
- 测评题库仍以内嵌代码结构为主，尚未抽成后台可配置数据模型

## 7. 代码核对结论

基于当前仓库代码，可以把项目状态归纳为：

- 完成的是“工程底座、多端联调、首版用户资料与多模块业务闭环”
- 部分完成的是“正式登录、内容体系、幸运体系、个人中心延展、后台配置化”
- 尚未完成的是“分享海报、商业化、完整后台、完整数据模型与服务拆分”

换句话说：

- 这个项目已经可以作为开发起点继续扩展
- 已具备一版可演示的业务原型
- 但还不能视为一版完整可上线产品
- 目前最适合作为“首版原型 + 后续扩展脚手架”
