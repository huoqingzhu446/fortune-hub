# Fortune Hub 接口开发文档

更新时间：2026-04-22  
接口前缀：`/api/v1`

## 1. 文档目标

本文件用于沉淀 `Fortune Hub` 的目标接口体系，并核对当前仓库已经落地的接口能力。

当前接口状态结论：

- 已完成：健康检查、移动端 / 管理端 dashboard、用户登录与资料、首页聚合、完整星座模块首版、八字解读、性格测评、情绪自检
- 部分完成：正式微信授权链路、幸运体系、内容管理化、后台运营接口
- 未完成：分享海报、订单 / 会员 / 广告、后台 CRUD、完整内容体系

## 2. 统一约定

### 2.1 请求头

客户端当前实际会透传：

- `Authorization: Bearer <token>`
- `X-Client: fortune-hub-mobile | fortune-hub-admin`

建议后续补充：

- `X-Request-Id: <uuid>`

### 2.2 响应结构

当前大部分业务接口已经统一为：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "timestamp": "2026-04-22T03:20:00.000Z"
}
```

例外：

- `GET /api/v1/health` 仍然直接返回对象

建议后续：

- 新增业务接口继续沿用 `code/message/data/timestamp`
- 若需要链路追踪，再补 `requestId`

### 2.3 鉴权策略

小程序端当前实际策略：

- 前端提交登录 `code`
- 后端通过微信 `jscode2session` 或开发环境 mock `openid`
- 服务端签发业务 token
- token 会话保存在 Redis，而不是 JWT
- 后续通过 `Authorization` 访问业务接口

管理端：

- 当前没有后台管理员登录和权限体系

### 3.2 用户与登录

#### `POST /api/v1/auth/wechat-login`

用途：

- 小程序端登录
- 接收微信 `code`
- 返回业务 token 与用户基础信息

请求体：

```json
{
  "code": "wx-login-code",
  "platform": "mp-weixin"
}
```

当前状态：已完成

说明：

- 已支持真实 `jscode2session`
- 未配置微信密钥时会回退到开发环境 mock `openid`

用途：

## 8. 实施建议

- 继续保持 `code/message/data/timestamp` 作为统一业务响应格式
- 业务结果类接口优先围绕 `records` 聚合，而不是为每个页面单独重造历史结构
- 题库与内容先配置化，再考虑后台可视化 CRUD
- 海报生成建议独立服务层，避免直接耦合在测评或八字模块中
