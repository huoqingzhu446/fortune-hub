# Fortune Hub 接口开发文档

更新时间：2026-04-23  
接口前缀：`/api/v1`

## 1. 文档目标

本文件用于沉淀 `Fortune Hub` 的目标接口体系，并核对当前仓库已经落地的接口能力。

当前接口状态结论：

- 已完成首版：健康检查、首页聚合、用户登录与资料、统一历史、星座模块、八字解读、性格测评、情绪自检、幸运体系首版、报告与海报、会员 / 订单 / 广告最小闭环、管理员登录、题库后台、内容中心与商业化配置
- 部分完成：正式微信授权链路、星座 / 幸运内容后台化、完整后台生命周期
- 未完成：`/settings`、独立幸运物库与模板中心、migration、正式支付验签与上线级风控

## 2. 统一约定

### 2.1 请求头

客户端当前实际会透传：

- `Authorization: Bearer <token>`
- `X-Client: fortune-hub-mobile | fortune-hub-admin`

建议后续补充：

- `X-Request-Id: <uuid>`

### 2.2 响应结构

当前大部分业务接口已经统一为：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "timestamp": "2026-04-23T03:20:00.000Z"
}
```

例外：

- `GET /api/v1/health` 仍然直接返回对象

### 2.3 鉴权策略

小程序端 / H5 当前实际策略：

- 前端提交登录 `code`
- 后端通过微信 `jscode2session` 或开发环境 mock `openid`
- 服务端签发业务 token
- token 会话保存在 Redis
- 后续通过 `Authorization` 访问业务接口

管理端：

- 已接入后台管理员登录、Redis 会话与 Bearer token
- `/api/v1/admin/question-bank/*`、`/api/v1/admin/fortune-contents/*`、`/api/v1/admin/membership-products/*`、`/api/v1/admin/ad-configs/*` 已接入管理员鉴权

- 返回业务 token 与用户基础信息

```json
{

}
```

当前状态：已完成首版

说明：

- 已支持真实 `jscode2session`
- 未配置微信密钥时会回退到开发环境 mock `openid`
- 前端已接 H5 开发环境登录与小程序 `uni.login` 登录入口

### 第三批

## 8. 实施建议

- 继续保持 `code/message/data/timestamp` 作为统一业务响应格式
- 结果类接口优先围绕 `records` 聚合，不要为每个页面单独重复造历史结构
- 已有后台接口应继续往“生命周期 + 发布流 + 审计”推进
- 海报生成建议单独抽服务层，不要直接耦合在测评或八字模块中
- 接口继续演进时，应同步维护 [开发文档](./开发文档.md) 与 [数据库设计文档](./数据库设计文档.md)
