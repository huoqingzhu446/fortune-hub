# Fortune Hub 数据库设计文档

更新时间：2026-04-22  
数据库类型：MySQL 8.x

## 1. 文档目标

本文件用于沉淀 `Fortune Hub` 的目标数据库结构，并核对当前仓库已经实现的数据层能力。

当前数据层结论：

- 已完成首版：MySQL 接入、TypeORM 接入、`users`、`records`、`fortune_contents`、`assessment_questions`、`assessment_test_configs`、`assessment_test_groups`、`assessment_sessions`
- 部分完成：用户资料、历史结果、幸运签内容、题库配置与运营分类首版持久化
- 未完成：配置、订单、分享、广告、幸运物内容库、模板、会员商品等核心表

## 2. 设计原则

目标设计原则：

- 优先使用 `snake_case`
- 所有业务主表建议使用 `bigint unsigned` 自增主键
- 金额字段使用 `decimal(10,2)`
- 状态字段优先使用 `varchar` 或 `tinyint`
- 扩展性较高的数据优先拆成独立表；结果快照和弹性结构可用 `json`

当前代码实际情况：

- TypeORM 当前仍以 `synchronize` 自动同步表结构为主
- 字段命名以实体定义为准，和理想化的 `snake_case` 设计稿有差异
- 如果准备进入正式环境，需要补 migration 与结构发布策略

## 6. 表关系建议

### 当前实际关系

```text
users
└─ 1:N records

fortune_contents
└─ 暂时独立使用，供首页幸运签和幸运签详情读取

assessment_test_groups
└─ 1:N assessment_test_configs

assessment_test_configs
└─ 1:N assessment_questions

assessment_sessions
└─ 早期示例表，未参与主业务闭环
```

### 目标扩展关系

```text
users
├─ 1:N orders
├─ 1:N records
└─ 1:N shares

records
└─ 1:N shares

assessment_test_groups
└─ 1:N assessment_test_configs

assessment_test_configs
└─ 1:N assessment_questions

fortune_contents
└─ 为 zodiac / lucky / bazi 提供内容来源

lucky_items
└─ 为幸运物推荐提供内容来源
```

## 10. 实施建议

- 不建议继续扩展 `assessment_sessions` 作为正式业务表
- 当前应优先把幸运体系内容、分享海报和商业化相关表补齐
- `records.resultData` 仍适合作为首版结果快照，但如果后续模板越来越复杂，建议拆出模板与海报快照表
- 若准备进入正式环境，应尽快从 `synchronize` 过渡到 migration
- 实体继续变化时，应同步维护 [开发文档](./开发文档.md) 与 [接口开发文档](./接口开发文档.md)
